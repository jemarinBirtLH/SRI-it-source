Historia de la Informática, Internet y el Estado Actual de las Redes

Desde los primeros ordenadores hasta la hiperconectividad actual, entender el pasado de las redes te ayuda a dominar el presente.

🔹 Los inicios (1940-1960)

Los primeros ordenadores eran máquinas aisladas y enormes (ENIAC, UNIVAC). A finales de los 50, conceptos como tiempo compartido y conmutación de paquetes (Paul Baran, Donald Davies) sentaron las bases de la comunicación digital.

🔹 ARPANET y el nacimiento de Internet (1969-1983)

En 1969, ARPANET conecta cuatro universidades estadounidenses. Se desarrolla el protocolo NCP y luego, en 1983, se adopta el TCP/IP (Vinton Cerf, Robert Kahn), el estándar que aún hoy usamos. La red crece y se separa la parte militar (MILNET).

🔹 La explosión de la Web y la comercialización (1990-2000)

Tim Berners-Lee crea la World Wide Web en el CERN (1989-1991). Navegadores como Mosaic y Netscape popularizan Internet. Se privatiza la espina dorsal y aparecen los primeros proveedores de acceso (ISP). El ancho de banda crece con el módem, luego ADSL.

🔹 Redes modernas y fibra óptica (2000-2010)

La fibra óptica despliega redes de alta capacidad. Aparecen estándares como Gigabit Ethernet, Wi‑Fi (802.11), MPLS y IPv6 para afrontar la escasez de direcciones. Las empresas adoptan arquitecturas de redes convergentes (voz, datos, vídeo).

🔹 La era de la hiperconectividad (2010-actualidad)

Hoy vivimos en un mundo IoT (Internet de las Cosas), con millones de sensores y dispositivos. El 5G promete latencias ultrabajas, la virtualización de redes (SD‑WAN, NFV) y el Edge Computing acercan el procesamiento a los usuarios. Las infraestructuras críticas dependen de redes robustas y seguras.

Comprender la evolución de las redes te sitúa en contexto para dominar tecnologías como subnetting, DHCP, DNS y las futuras innovaciones.

Parte IV: SSH y Acceso Remoto

19. ¿Qué es SSH?

SSH (Secure Shell) es un protocolo que permite abrir una consola remota cifrada contra un equipo (router, switch, servidor…). Nació para sustituir a Telnet, que enviaba usuario y contraseña en texto plano.

A mediados de los 90, Tatu Ylönen desarrolló SSH para proteger accesos remotos en redes inseguras. Desde entonces se ha convertido en el estándar para administrar sistemas Linux, routers y equipos de red a través de Internet o redes corporativas.

  • Puerto típico: 22/TCP (puede cambiarse por seguridad/política).
  • Autenticación: usuario/contraseña o claves públicas (llave privada + llave pública).
  • Usos: administración remota, túneles cifrados (port forwarding), copia de ficheros (SCP/SFTP).
Idea clave: con SSH tienes una consola como si estuvieras delante del equipo, pero de forma cifrada.
20. PuTTY: cliente SSH para Windows

PuTTY es uno de los clientes SSH más usados en Windows. Permite conectarse por SSH, Telnet, Serial, etc., a routers, switches y servidores.

Descarga oficial: Página oficial de PuTTY

Uso básico de PuTTY
  1. Instala PuTTY desde la web oficial (versión Windows 64‑bit recomendada).
  2. Abre PuTTY.
  3. En Host Name (or IP address), pon la IP o nombre del equipo (ej. 192.168.1.1).
  4. En Port, deja 22 salvo que el dispositivo use otro puerto.
  5. En Connection type, selecciona SSH.
  6. Opcional: en Saved Sessions, escribe un nombre (ej. router-cisco-lab) y pulsa Save para no reescribir datos cada vez.
  7. Pulsa Open, acepta la clave del host la primera vez y luego introduce usuario y contraseña.
# Ejemplos de sesiones típicas en PuTTY Host: 192.168.1.1 Port: 22 Connection type: SSH (router Cisco) Host: 192.168.1.2 Port: 22 Connection type: SSH (router Teldat) Host: 192.168.88.1 Port: 22 Connection type: SSH (router MikroTik)
21. Comandos SSH desde terminal

En Linux, macOS o Windows con OpenSSH instalado puedes usar SSH directamente desde la consola:

# Conexión básica ssh usuario@192.168.1.1 # Especificar puerto (ej. 2222) ssh -p 2222 usuario@192.168.1.1 # Conexión con clave privada ssh -i ~/.ssh/id_rsa usuario@192.168.1.1

Para usar claves, primero generas el par en tu equipo (por ejemplo, con ssh-keygen) y registras la clave pública en el dispositivo al que te conectas.

Buenas prácticas: cambia usuarios por defecto, usa contraseñas robustas, restringe el acceso SSH por IP y, si es posible, usa autenticación por clave pública.
23. Buenas prácticas de uso de SSH
  • Desactiva Telnet cuando puedas; deja solo SSH para administración remota.
  • Crea usuarios específicos para administración en lugar de usar siempre el admin por defecto.
  • Restringe el acceso con ACL o firewall (por ejemplo, solo desde la red de administración).
  • Cambia el puerto por defecto si la política de la empresa lo permite y documenta el cambio.
  • Usa claves públicas en entornos críticos en vez de solo usuario/contraseña.
  • Documenta IP, puerto, usuarios y cambios hechos en cada router/servidor.
# Ejemplo (muy genérico) de pasos en un router: 1) Habilitar SSH y crear usuario 2) Deshabilitar Telnet en líneas VTY 3) Limitar acceso SSH por ACL / firewall 4) Probar acceso desde red de gestión 5) Documentar cambios
24. AnyDesk y asistencia remota

A veces, además de consola o SSH, necesitas ver el escritorio del usuario. Para eso se usan herramientas de asistencia remota como AnyDesk.

Descarga AnyDesk: Página oficial de descarga de AnyDesk

Cómo recibir soporte (usuario final)
  1. Descarga e instala AnyDesk desde la página oficial.
  2. Abre AnyDesk y localiza tu dirección AnyDesk (código numérico o alfanumérico).
  3. Comparte ese código solo con la persona o empresa que te va a dar soporte.
  4. Cuando aparezca la solicitud de conexión, revisa el nombre del equipo remoto y pulsa Aceptar.
  5. Al terminar, cierra AnyDesk o revisa que no queden permisos activos que no necesites.
Cómo dar soporte remoto (técnico)
  1. Pide al usuario que instale/abra AnyDesk y te indique su dirección.
  2. En tu AnyDesk, escribe esa dirección y pulsa Conectar.
  3. Espera a que el usuario acepte la conexión y los permisos de control de teclado/ratón.
  4. Explica siempre qué vas a hacer (cambios en red, instalación de software, etc.).
  5. Al finalizar, cierra la sesión y verifica con el usuario que todo funciona correctamente.
Seguridad: no guardes contraseñas en texto plano durante una sesión remota, pide autorización explícita para acceder a datos sensibles y respeta siempre la política de la empresa/cliente.

Parte I: Subnetting y Planificación

1. ¿Qué es una red?

Una red es un conjunto de dispositivos interconectados que se comunican utilizando direcciones IP. Estos dispositivos pueden ser ordenadores, servidores, routers, switches, etc.

PC / Host Switch Router Internet
2. Direcciones IP y Máscara de Subred

Una dirección IPv4 está compuesta por 4 octetos (32 bits). La máscara de subred determina qué parte identifica la red y qué parte el host.

Ejemplo: 192.168.1.10 /24 ? primeros 24 bits (3 octetos) red, últimos 8 bits host.
  • IP Identificador único de 32 bits
  • Máscara Define red y host
  • CIDR Notación /n (ej. /24)
# Ejemplo en binario IP: 192.168.1.10 = 11000000.10101000.00000001.00001010 Máscara: 255.255.255.0 = 11111111.11111111.11111111.00000000 Red: 192.168.1.0 = 11000000.10101000.00000001.00000000
3. Concepto de Subnetting

División de una red grande en subredes más pequeñas para optimizar direcciones y mejorar gestión.

Ventajas: optimización, organización, aislamiento de tráfico, reducción de broadcast.
Red original /24 Subred 1 /26 Subred 2 /26 Subred 3 /26 Subred 4 /26

Parte II: Protocolo DHCP

7. Parámetros de red

Un equipo necesita: IP, máscara, gateway, DNS. Configuración puede ser estática o dinámica (DHCP).

Estática
  • Control total
  • IP fija para servidores
  • Riesgo de conflictos
Dinámica (DHCP)
  • Configuración automática
  • Sin conflictos
  • Gestión centralizada
# Ejemplo /etc/network/interfaces (Debian) auto enp0s3 iface enp0s3 inet static address 192.168.1.100 netmask 255.255.255.0 gateway 192.168.1.1 # DHCP iface enp0s3 inet dhcp
8. ¿Qué es DHCP?

DHCP (Dynamic Host Configuration Protocol) permite a dispositivos obtener configuración de red automáticamente.

  • Modelo cliente-servidor
  • Puertos UDP: 67 (servidor), 68 (cliente)
  • Basado en BOOTP (RFC 1541, 2131)
9. Tipos de asignación
Manual/Estática

Reserva IP por MAC. Ideal para servidores.

Automática

Asigna IP permanente del pool.

Dinámica

IP temporal (lease time). La más común.

10. Proceso DORA
1. DISCOVER

Cliente broadcast

2. OFFER

Servidor ofrece IP

3. REQUEST

Cliente acepta

4. ACK

Servidor confirma

Tiempos de concesión: T1 (50%) renovación con mismo servidor; T2 (87.5%) con cualquier servidor; expiración ? nuevo DISCOVER.

12. Parámetros asignados
  • IP, máscara, gateway, DNS, nombre de dominio, broadcast, lease time, etc.
13. DHCP Relay Agent

Permite que un servidor DHCP atienda múltiples subredes. El router convierte broadcast en unicast hacia el servidor.

# Instalación en Debian sudo apt install isc-dhcp-relay # Configurar /etc/default/isc-dhcp-relay SERVERS="192.168.1.100" INTERFACES="enp0s3 enp0s8"
14. Configuración del servidor DHCP (isc-dhcp-server)
# Instalación sudo apt install isc-dhcp-server # Archivo /etc/dhcp/dhcpd.conf authoritative; default-lease-time 600; max-lease-time 7200; option domain-name "miempresa.local"; option domain-name-servers 8.8.8.8, 8.8.4.4; subnet 192.168.10.0 netmask 255.255.255.0 { range 192.168.10.50 192.168.10.100; option routers 192.168.10.1; } host servidor-web { hardware ethernet 00:1A:2B:3C:4D:5E; fixed-address 192.168.10.10; }

Gestión del servicio:

sudo systemctl start isc-dhcp-server sudo systemctl enable isc-dhcp-server sudo tail -f /var/log/syslog
